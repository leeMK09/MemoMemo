## nginx 성능 최적화

- nginx 가 응답을 보낼 때 비용이 드는 지점은 크게 네가지
- 첫 번째는 메모리 복사 비용
  - 파일을 디스크에서 읽고, 커널 버퍼로 올리고, 다시 사용자 공간으로 복사하고, 다시 소켓 버퍼로 복사하는 과정은 CPU 를 사용한다.
  - 파일이 작을 때는 체감이 적지만, 정적 파일이 많아지고 트래픽이 커지면 이 복사 비용이 꽤 큰 차이를 만든다
- 두 번째는 패킷 분할과 전송 비용
  - 같은 데이터를 보내더라도 자잘한 조각으로 쪼개서 보내면 패킷 수가 늘어나고, 네트워크 스택이 처리해야 할 작업이 증가한다
  - 반대로 너무 많이 모아 보내려 하면 첫 응답이 늦어질 수 있고 결국 이 부분은 처리량과 지연이 해결해야할 문제이다
- 세 번째는 TCP 연결을 맺고 끊는 비용
  - HTTP 요청 하나 처리할 때마다 TCP 연결을 새로 열면 3-way handshake 가 매번 일어나고, HTTPS 라면 TLS 핸드셰이크 비용도 추가된다
  - 이 비용은 서버 CPU 뿐 아니라 클라이언트 체감 속도에도 직접 영향을 준다
- 네 번째는 HTTP 레벨에서의 불필요한 데이터 전송과 왕복 비용
  - 헤더가 과도하게 크거나, 캐시를 못 써서 매번 본문을 다시 보내거나 압축이 비효율적으로 걸리거나, 커넥션 재사용이 안되면 nginx 가 아무리 빨라도 전체 체감 성능은 떨어질 수밖에 없다

</br>

### sendfile

- `sendfile` 은 특히 정적 파일을 전송할 때 CPU 를 덜 쓰게만드는 방향의 최적화이다
- 보통 nginx 가 정적 파일을 응답으로 보낼 때, 가장 단순한 방식은 파일을 읽어서 nginx 프로세스가 들고 있다가 다시 소켓으로 쓰는 방식이다
- 이 구조에서는 파일 데이터나 디스크에서 복사를 수행한 뒤, 커널 공간과 사용자 공간 사이를 몇 번 오가게 된다
- 그런데 `sendfile` 을 `on` 하면 nginx 는 파일 내용을 굳이 사용자 공간으로 가져오지 않고 커널이 파일 데이터와 소켓 전송을 최대한 직접 처리하라는 방향으로 동작하게 된다
- 이를 활용하면 정적 파일 서빙이 많을수록 CPU 절감 효과를 기대할 수 있고, 큰 파일일수록 이점이 커질 수 있다
  - 이미지, CSS, JS, 폰트 등
- 하지만 단점은 동적 응답에는 직접적인 이득이 없다 그리고 특정 파일시스템이나 네트워크 파일시스템, 오래된 커널 환경, 일부 가상화 조합에서는 예상과 다르게 동작할 수 있다

</br>

### tcp_nopush

- `tcp_nopush` 는 `sendfile` 과 같이 사용해야 한다
  - 둘 다 정적 파일 전송 경로에서 의미가 크기 때문이다
- nginx 가 응답을 만들면 그 응답은 한 번에 거대한 덩어리로 전송되는 것이 아닌 TCP 스택이 적당한 크기의 세그먼트로 나누고 소켓 버퍼를 거쳐 네트워크 패킷으로 흘러간다
- 문제는 이 과정에서 HTTP 응답 헤더와 파일 본문이 너무 잘게 나뉘어 나가면 비효율이 생긴다
  - 예를 들어 헤더만 먼저 작게 전송되고, 파일 바디도 작게 끊어서 여러 패킷으로 흩어지면 패킷 수가 늘고 커널과 NIC 가 처리해야 하는 작업이 많아진다
- `tcp_nopush` 는 이 상황에서 nginx 가 지금 당장 작게 전송하지 말고, 어느 정도 모아서 효율적인 크기로 보내라는 방향으로 동작하게 돕는 설정이다
  - 리눅스 관점에서는 `TCP_CORK` 와 관련된 개념이다
- 이 옵션은 결국 처리량과 패킷 효율을 높이는 것이며 첫 응답을 무조건 가장 빨리 보내겠다는 관점이 아닌 정적 파일처럼 덩어리가 있는 응답에서 헤더와 바디를 보다 효율적인 패킷 구성으로 실어 보내는 것이 핵심이다
- 그래서 `tcp_nopush` 는 `sendfile` 과 함께 묶어 설명되기도 한다
  - `sendfile` 이 사용자 공간 복사를 줄이는 최적화라면 `tcp_nopush` 는 네트워크로 보내는 패킷 구성을 좀 더 효율적으로 만드는 최적화
  - 둘은 같은 문제를 해결하는 것이 아닌 정적 파일 전송이라는 같은 흐름 안에서 서로 다른 비용 지점을 줄이는 역할을 수행
- 하지만 이 옵션의 단점은 모아서 보내라는 전략을 사용하며 결국 즉시성을 일부 포기하는 전략이다
  - 즉 첫 바이트를 최대한 빨리 보내야 하는 인터렉티브한 응답이나, 작은 응답을 재빠르게 던지는 API 에서는 오히려 지연을 키우는 방향으로 작용할 수 있다
- 실무에서는 보통 nginx 가 정적 파일을 서빙하는 경로에서는 `sendfile` 과 함께 `tcp_nopush` 를 켜는 방향을 많이 검토한다 또한 CDN 이 앞단에 있다면, 정적 파일 전송 최적화의 상당 부분을 CDN 이 가져가기 때문에 원본 nginx 에서 이 옵션의 체감 효과가 줄어들 수 있다

</br>

### tcp_nodelay

- `tcp_nodelay` 는 Nagle 알고리즘을 끄는 옵션이다
- Nagle 알고리즘은 네트워크 효율을 위해 작은 데이터를 모아 보냈다가 한 번에 전송하는 전략이다
  - 즉 자잘한 패킷을 바로바로 보내지 않고 조금 쌓아서 보내자는 방식
- 이 전략은 네트워크 효율에는 좋을 수 있어도 사용자 입장에서는 지연으로 느껴질 수 있다는 문제가 있다
- `tcp_nodelay` 를 켜면 nginx 는 이런 작은 데이터를 모았다가 보내지 말고 바로 보내라는 쪽으로 동작하게 된다
  - 즉 이 옵션의 핵심은 패킷 효율보다 응답 지연 감소를 우선시하는 것이다
- 여기서 헷갈리는 지점은 `tcp_nopush` 는 모아서 보내는 방식이고 `tcp_nodelay` 는 바로 보내는 방식이니 둘은 완전히 충돌하는 것 아닌가 하는 의문이 들 수 있다
  - 이를 같은 문제를 두고 충돌되는 해결 방안으로 보면 안되고 각 상황에 알맞는 방식이 있다라고 봐야한다
  - 정적 파일 전송 초기 구간에서는 nginx 가 헤더와 바디를 더 효율적으로 묶어 보내기 위해 `tcp_nopush` 성격의 이점을 활용할 수 있다
  - 반대로 응답이 끝나고 `keep-alive` 상태에서 후속 요청이나 작은 추가 응답을 처리할 때는 `tcp_nodelay` 가 작은 데이터를 지연 없이 내보내는데 유리하다
  - 즉 둘은 단순히 둘 중 하나만 켜야하는 옵션보다는 응답 흐름의 다른 단계에서 다른 최적화 목적을 갖는 옵션으로 보는 것 이다
  - 정적 파일 위주의 서버, 특히 큰 파일이나 많은 리소스를 제공하는 웹 서버라면 `tcp_nopush` 의 이점을 더 잘 볼 수 있다
  - 반대로 API 서버, 작은 JSON 응답이 잦은 서비스, 실시간성이 중요한 응답은 `tcp_nodelay` 의 이점이 크다

</br>

### keepalive_timeout

- HTTP/1.1 에서는 Keep-Alive 를 통해 하나의 TCP 연결을 여러 요청에 재사용할 수 있다
  - 만약 브라우저가 파일을 받을 때마다 매번 TCP 연결을 새로 맺는다면 매 요청마다 3-way-handshaker 가 반복되며 불필요한 왕복과 CPU 비용이 계속 발생한다는 의미이다
- `keepalive_timeout` 은 마지막 요청 이후 이 TCP 연결을 얼마 동안 유지할지 정하는 설정이다
  - 즉 한번 연결된 클라이언트를 빨리 끊지말고 일정 시간 동안은 같은 연결을 재사용할 기회를 주자라는 설정이다
- 이 설정의 장점은 핸드셰이크 비용이 줄고, 같은 브라우저가 이어서 보내는 후속 요청을 더 빨리 처리할 수 있다
- 하지만 Keep-Alive 는 연결을 오래 유지하므로 그 연결이 계속 리소스를 잡는다는 문제가 있다
  - TCP 연결 하나는 실제로 소켓, 메모리, 파일 디스크립터, 커널 상태를 점유한다
- 이 문제를 여러 상황에서 효율적으로 사용해야 한다
  - 첫 째, 클라이언트와 업스트림을 분리해서 생각해야 한다
    - 클라이언트와 nginx 사이 Keep-Alive, 그리고 nginx 와 애플리케이션 서버 사이 Keep-Alive 는 목적이 다를 수 있다
  - 둘 째, CDN 이나 LB 계층의 역할 분담
    - 클라이언트 연결은 CDN 이나 ALB 가 상대적으로 더 많이 트래픽을 받고, 원본 nginx 는 더 짧고 타이트하게 가져가는 구조
  - 셋 째, 실제 유휴 연결 수와 신규 연결 처리 이력을 참고
    - 단순 latency 만 보고 keepalive_timeout 을 길게 잡으면, 피크 타임에 오히려 병목 현상이 발생한다

</br>

### keepalive_requests

- `keepalive_requests` 는 하나의 Keep-Alive 연결에서 몇 개 요청까지 처리한 뒤 연결을 닫을지 정하는 설정이다
- 이 설정이 존재하는 이유는 연결 재사용도 결국 무한정 좋은 것은 아니기 때문이다
  - 하나의 연결을 너무 오래 살려두면, 그 연결에 연결된 내부 상태와 버퍼, 소켓 자원이 장기간 유지되며 일부 연결이 지나치게 오래 살아남아 리소스를 점유하는 상황이 생긴다
- `keepalive_requests` 는 이 정도 요청을 처리했으면 이 연결은 충분히 사용했으니 닫고 다시 새 연결을 갱신하자는 일종의 정리 장치이다
- 장점은 장기 연결이 자원을 붙잡고 있는 문제를 완화할 수 있고, 오래된 연결을 적절한 시점에 정리할 수 있다 그리고 지나친 연결 독점을 막는 균형을 잡을 수 있다
- 반대로 값을 너무 낮게 잡으면 Keep-Alive 의 장점을 잃게된다
- 실무에서는 이 값을 단독으로 보기보다 `keepalive_timeout` 과 같이 봐야한다
  - 연결 수명은 시간 기준으로도 제한되고, 요청 수 기준으로도 제한되기 때문에 얼마나 오래 연결할 것 인가 다른 하나는 얼마나 많이 사용할 것인가를 제어한다
