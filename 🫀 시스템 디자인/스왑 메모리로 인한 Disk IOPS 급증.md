## 스왑 메모리로 인한 Disk IOPS 급증

- OS는 프로세스가 사용하는 메모리를 관리할 때 가상 메모리(Virtual Memory) 구조를 사용한다
    - 애플리케이션 프로세스는 자신이 매우 큰 메모리를 사용하는 것처럼 보이지만, 실제로는 물리 메모리(RAM)와 디스크에 있는 스왑 공간을 함께 사용하여 메모리를 관리한다
- 리눅스 커널은 시스템 메모리가 부족해지기 시작하면 사용 빈도가 낮은 메모리 페이지를 RAM에서 제거하고 디스크의 swap 영역으로 이동시키는 작업을 수행한다
    - 이 작업을 swap-out 이라고 부른다
- 이때 커널은 다음과 같은 작업을 수행한다
    - OS 커널이 특정 프로세스의 메모리 페이지를 선택하고 그 페이지의 내용을 디스크에 있는 swap 공간에 기록
    - 이 작업은 결국 디스크 write opration 이 되기 때문에 스토리지 장치에서는 Disk Write IOPS 가 증가하는 현상이 나타난다
    - 이후 해당 메모리 페이지가 다시 필요해지면 커널은 swap 영역에 저장된 데이터를 다시 RAM 으로 읽어온다
    - 이 과정은 swap-in 이라고 부르며, 이때는 디스크에서 데이터를 읽어와야 하므로 Disk Read IOPS 가 증가한다
- 즉 메모리 부족 상황에서 스왑이 발생하면 디스크에서는 다음 두 종류의 I/O 가 발생한다
    - OS 가 RAM 에서 밀려난 메모리 페이지를 디스크로 기록하는 write opration 이 발생하고, 이후 프로세스가 그 데이터를 다시 필요로 하면 디스크에서 메모리 페이지를 읽어오는 read opration 이 발생한다
    - 이 두 작업이 반복되면 디스크에서는 매우 많은 I/O 요청이 발생하게 되고, 결과적으로 Disk IOPS 가 급격히 증가하는 현상이 나타난다

</br>

## Thrashing

- 위 상황이 심해지면 **Thrashing** 이라는 상태가 발생할 수 있다
- Thrashing 은 OS 가 실제 애플리케이션 작업을 수행하는 대신 메모리 페이지를 RAM 과 swap 사이에서 계속 이동시키는 작업에 대부분의 시간을 사용하게 되는 상태를 의미한다
    - OS 가 RAM 에 없는 페이지를 디스크에서 읽어오는 read I/O 작업을 수행하고, 동시에 다른 페이지를 swap 영역으로 내보내기 위해 write I/O 작업을 수행한다
    - 이 과정이 반복되면 디스크에서는 초당 매우 많은 I/O 요청이 발생하게 된다

</br>

## 해결 방안들

1. 단순히 물리 메모리를 증가시킨다
    - 메모리 부족이 표면적인 원인이기 때문에 RAM 을 확장
2. 애플리케이션 메모리 사용량을 줄인다
    - 예를들어 JVM 애플리케이션이라면 heap 크기를 조정하거나 캐시 정책을 수정하여 메모리 사용량을 줄일 수 있다
3. swap 사용 정책을 조정
    - 리눅스에서는 `vm.swappiness` 라는 커널 파라미터가 존재한다
    - 이 값은 커널이 얼마나 적극적으로 swap 을 사용할지 결정한다
    - 일반적으로 서버 환경에서는 이 값을 낮게 설정하여 swap 사용을 최소화한다
    - `sysctl vm.swappiness=10`

## 번외) 스왑 메모리로도 부족하여 서버가 죽으면

- 백엔드 서버 : nginx 가 요청을 받음, 이후 express.js 에게 요청을 전달
- 여기서 가장 높은 시나리오는 I/O backlog + Node event loop 지연 + 요청 backlog 증가가 결합된 상황
- 처음에 디스크 병목(스왑 메모리로 인해) 때문에 요청 처리가 지연되면 Nginx 는 upstream인 Express 서버로 요청을 전달하려고 시도한다
- 하지만 Node.js 는 싱글 이벤트 루프 구조이기 때문에 디스크 I/O 대기나 GC 지연이 발생하면 요청 처리가 늦어진다
- 이때 다음과 같은 현상이 발생한다
- Nginx 는 처리되지 않은 요청을 connection backlog 나 worker queue 에 유지한다
    - 동시에 클라이언트에서도 retry 요청이 발생할 수 있다
- 이렇게 되면 서버 내부에서는 이미 처리되지 못한 요청이 상당히 쌓여 있게 된다
- 만약 EBS 를 순간적으로 확장하면 이후 디스크 I/O 가 잠시 정상화되며 Node.js 이벤트 루프가 다시 실행되기 시작된다
- 이 순간 Node는 backlog 에 있던 요청들을 다시 처리하기 시작한다
- 문제는 이 시점에서 이미 요청 수가 서버 처리 능력을 초과한 상태일 가능성이 높다
- 이때는 Node.js 프로세스가 backlog 요청을 처리하면서 다시 많은 메모리를 할당하게 된다
    - GC 가 반복적으로 발생한다
    - 메모리 사용량이 다시 증가한다
    - swap 이 다시 발생한다
    - 디스트 I/O 가 다시 증가한다
- 이렇게 되면 시스템은 다시 thrashing 상태로 들어가면서 서버가 다시 다운될 수 있다
