## PM2

- 노드 기반 백엔드 서비스는 싱글 스레드로 운영되며 I/O Bound 작업 시 이벤트 루프를 통해서 처리된다
- 이런 환경에서 멀티 프로세스 기반 운영을 하기도 하는데 이를 좀 더 쉽게 운영하기 위한 프로세스 매니저가 `PM2` 이다
- PM2 는 단순히 `node app.js` 를 대신 실행해주는 도구가 아니다
- 정확히는 Node 애플리케이션 프로세스를 백그라운드에서 지속적으로 감시하고, 죽으면 다시 띄우고, 여러 개를 묶어서 관리하고, 로그와 배포 운영 기능까지 제공하는 프로세스 매니저이다
- 즉 PM2 본체도 하나의 데몬 프로세스라는 점이다
    - 즉 사용자가 터미널에서 `pm2 start app.js` 를 실행하면 그 순간 Node 앱이 바로 터미널에 붙어서 도는 것이 아닌, PM2 데몬이 앱 프로세스를 자식 프로세스로 띄우고 관리 목록에 등록한다
    - 이후에는 쉘 세션을 끊어도 앱이 계속 살아있고, PM2 가 그 생명주기를 추적한다

</br>

### OS 관점에서 PM2

- 먼저 OS 위에 PM2 데몬이 띄워져 있다
- 그 다음 PM2 데몬이 사용자가 등록한 각 앱을 별도 프로세스로 생성한다
- 이때 앱이 하나면 자식 프로세스 하나, 클러스터 모드면 같은 앱 코드로 여러 자식 프로세스를 만든다
- 그리고 PM2 는 이 자식 프로세스들의 PID, 상태, 재시작 횟수, CPU/메모리 사용량, 로그 파일 경로 같은 메타데이터를 계속 들고 있는다
    - `pm2 ls`, `pm2 monit`, `pm2 logs` 로 이 상태를 관찰할 수 있고, CPU/메모리 모니터링도 제공한다
- 즉 `node` 프로세스가 주체가 아니라 PM2 데몬이 주체이다
- 앱이 비정상 종료되면 OS 가 자식 프로세스 종료를 PM2 에 알려주고, PM2 는 등록된 정책에 따라 다시 띄운다
- 그래서 PM2 를 단순 실행기가 아니라 감시자(supervisor) 로 이해해야 한다

</br>

### 장애가 났을때 PM2 는 어떻게 복구하는가

**애플리케이션 예외**

- 예를 들어 Express 서버가 처리 중 `unhandled exception` 으로 프로세스가 종료되면 PM2 는 그 종료를 감지하고 자동 재시작한다
- 여기서 중요한 것은 PM2 가 프로세스를 재시작해줄 뿐 애플리케이션 상태를 복구해주는 것은 아니다
    - 예를 들어 메모리에 들고 있던 세션, in-memory queue, 요청 중간 상태는 날아간다
    - 실무에서는 외부 자원을 통해 복구 가능한 형태로 설계한다

**메모리 누수**

- PM2 는 메모리 한계를 넘으면 재시작하도록 설정할 수 있다
    - `--max-memory-restart` 또는 ecosystem 설정의 `max_memory_restart` 로 메모리 임계치를 걸 수 있다
- 이 기능은 메모리 누수 문제가 발생하면 서버 전체가 터지기 전에 프로세스를 강제로 순환시키는 장치이다
- 즉 원인 제거가 아닌 피해 축소의 개념

**계속 죽고 바로 다시 뜨는 크래시 루프**

- 예를 들어 DB 비밀번호가 틀렸거나 필수 환경변수가 빠져 있으면 앱은 뜨자마자 죽고, PM2 는 다시 띄우고, 또 죽고 반복하게 된다
    - 이때 재시작 간격이 촘촘하면 CPU 를 태우고 로그를 폭증시킨다
- PM2 는 이를 위해 `--restart-delay` 옵션과 `--exp-backoff-restart-delay` 전략을 제공한다
- 즉 첫 번째 시작은 1초 뒤, 그 다음은 2초, 그 다음은 4초처럼 백오프를 걸어서 장애 폭주를 어느정도 방어할 수 있다 → 2차 장애 유발 방지

**서버 재부팅 후 복구**

- PM2 는 `pm2 startup` 으로 부팅 시 데몬과 등록된 프로세스를 다시 살리는 절차를 제공한다
- 다만 여기서 오해 소지는 OS 가 올라온 뒤 PM2 가 올라오고 PM2 가 저장된 프로세스 목록을 다시 복원하는 것까지만 제공하며 애플리케이션 자체가 스스로 살아나는 것은 아니다
- `pm2 save` 로 현재 프로세스 리스트를 저장해두기도 한다

</br>

### PM2 실행 모드

**fork 모드**

- 가장 단순한 모드이며 앱 프로세스를 하나만 띄우는 형태이다
- 즉 `node app.js` 를 백그라운드에서 감시형으로 돌리는 것에 가깝다
- 이 모드는 구조가 단순하며 디버깅이 쉽다
- 로그를 볼 때도 이 요청이 어느 프로세스에서 처리되었는지 고민할 필요가 없다
- 백그라운드 워커, 스케줄러, 큐 컨슈머처럼 포트를 공유할 필요가 없는 프로세스에는 fork 모드가 자연스럽다
- 하지만 HTTP 서버에 트래픽이 많고 CPU 코어가 여러 개라면 fork 모드는 해당 서버의 멀티 코어를 다 사용하지 못한다
    - 이유는 프로세스가 하나이며 결국 메인 이벤트 루프 또한 하나이다

**cluster 모드**

- cluster 모드는 PM2가 같은 앱을 여러 인스턴스로 띄우고, 같은 서버 포트를 공유하게 해준다
    - `pm2 start api.js -i max` 로 전체 CPU 를 활용하는 클러스터를 만들 수 있고, built-in load balancer 및 zero-downtime reload 를 제공한다고 한다
    - 즉 서버에 코어가 8개면 `-i max` 로 8개의 워커를 만들 수 있다
- 장점은 첫째, 한 서버 안에서 CPU 코어를 훨씬 더 잘 활용할 수 있다. 둘째, 워커 하나가 죽어도 나머지 워커가 계속 받기 때문에 가용성이 올라간다 셋째, `pm2 reload` 로 무중단에 가까운 교체가 가능하다
- 하지만 단점도 존재한다
- 첫째, 상태 공유가 없다. 각 워커는 별도 프로세스이므로 메모리를 공유하지 않고 그래서 세션을 메모리에 저장하면 워커마다 다른 값을 보게 된다
- 둘째, 로그와 디버깅이 복잡해진다. 같은 앱 이름 아래 인스턴스가 여러 개이므로 같은 시각에 여러 워커 로그가 섞이면 원인 추적이 어려워진다
- 셋째, CPU 바운드 문제를 근본적으로 해결하지 못한다. 클러스터는 한 서버 안에 프로세스를 여러개 띄워줄 뿐, 각 워커 안의 이벤트 루프는 여전히 CPU 바운드 작업에 취약하다

**상황에 따른 선택**

- HTTP API 서버라면 단일 VM/EC2 환경에서는 cluster 모드가 꽤 유용하다
- 반면 큐 워커, 크론 잡, 배치, 순차 처리 작업은 fork 모드가 더 자연스럽다

</br>

### reload, restart, gracefulReload

- PM2 CLI 는 `reload`, `restart`, `gracefulReload` 를 구분한다
- `restart` 는 말 그대로 현재 프로세스를 죽이고 다시 띄운다
    - 짧더라도 연결 끊김이 생길 수 있고, 인플라이트 요청이 손실될 수 있다
- `reload` 는 특히 HTTP/HTTPS 앱에서 기존 워커를 한꺼번에 죽이지 않고 새 워커를 먼저 띄운 뒤 순차 교체하는 방식이다
    - 그래서 무중단에 가까운 처리를 할때 적합하다
- `gracefulReload` 는 앱에 `shutdown` 메시지를 보내 현재 연결을 스스로 정리할 기회를 준다
    - 실무에서는 HTTP 서버에 keep-alive 연결, 장시간 업로드, websocket, DB transcation 이 석여 있으면 단순 restart 보다 graceful 종료를 구현하는 편이 안전하다

</br>

### PM2 로그

- PM2 는 애플리케이션의 stdout/stderr 를 자동으로 훅킹해서 로그를 저장한다
- 기본 로그 경로는 `$HOME/.pm2/logs` 이고 `pm2 logs`, `pm2 flush` 등으로 조회/비우기가 가능하다
- 보통 표준 출력 로그 파일 / 에러 로그 파일 각각 따로 생긴다
- 하지만 로그 파일로 처리되기 때문에 디스크에 파일이 계속 쌓이게 된다
- 클러스터 모드의 경우 워커별 파일이 여러 개 생길 수 있다
- 여러 서버/인스턴스가 생기면 그 서버 안의 파일로 흩어진다
- 즉 기본 PM2 로그는 로컬 디스크 중심이기 때문에 소규모에는 충분하지만 분산 환경에서는 금방 한계가 온다

</br>

### 클러스터 모드에서의 로그

- cluster mode 에서는 각 클러스터 인스턴스가 자체 로그 파일을 가질 수 있다
    - 또는 `merge_logs: true` , `--merge-logs` 로 파일로 합칠 수 있다
- 프로세스 별로 로그를 분리하는 경우
    - 장점은 디버깅이다
    - 예를 들어 worker 3만 메모리 급증이 있었다면 그 프로세스의 로그만 집중해서 볼 수 있다, 장애 격리도 좋다
    - 단점은 운영이 힘들다
    - 워커가 8개면 out/error 파일이 16개 이상 생길 수 있고 재시작/스케일링까지 겹치면 파일 수는 금방 많아지게 된다
- `merge_logs` 로 합칠 경우
    - 조회가 편리하다 앱 전체 로그를 한 파일로 볼 수 있게된다
    - 단점은 동시성이 높은 앱에서 로그가 뒤섞일 수 있다는 점이다
    - 어느 요청이 어느 worker 에서 처리되었는지 구분이 어려워지고 merge 를 쓰더라도 로그 본문에 `request id`, `trace id` 같은 값을 반드시 널어야 한다
- 즉 merge 여부는 편의성과 추적성의 트레이드 오프이다

</br>

### logrotate

- PM2 는 기본적으로 로그를 계속 append 한다
- 그러면 시간이 지날수록 로그 파일은 계속 커지고 특히 API 서버, 에러 루프, 디버그 로그 남발, 요청 바디 로깅이 있으면 파일이 순식간에 GB 단위로 커질 수 있다
    - 디스크 용량 부족, 로그 읽기와 백업이 느려짐, 크래시 루프가 나면 짧은 시간에 수십만 줄이 생김
- `pm2-logrotate` 는 이런 상황에서 일정 크기나 일정 시점이 되면 현재 로그 파일을 잘라서 보관 파일로 넘기고, 새 파일을 다시 시작하게 해준다
- 장점은 디스크 폭주를 방지하고 오래된 로그를 보존 개수 기준으로 잘라낼 수 있다 그리고 운영자가 수동으로 로그 파일을 삭제하지 않아도 된다
- 하지만 이것도 한계가 있다
    - logrotate 는 파일 크기를 관리하는 기능이지 로그 분석 기능이 아니다
    - 즉 회전된 파일은 여전히 서버 로컬 디스크 위 파일이며 서버가 사라지면 같이 사라질 수 있다

</br>

### PM2 운영 문제점들

**reload 처리 시 요청이 끊김**

- 원인은 앱이 종료 시그널을 받아 graceful shutdown 을 하지 않기 때문이다
- 대응은 `SIGINT` / `SIGTERM` 또는 PM2 gracefulReload 시나리오에 맞춰 새 요청 수신을 중지하고 기존 연결을 drain 하는 코드를 넣는 것 이다

**클러스터 모드시 세션이 꼬임**

- in-memory session 의 경우 워커 간 메모리 공유를 하지 못해 세션이 끊긴다

**CPU 바운드 작업으로 인한 과부하**

- PM2 가 네트워크 병렬성은 올려도 CPU 바운드 연산 자체는 최적화해주지 않는다
- 대응은 작업을 분리하여 queue worker, worker_threads, 별도 배치 서비스 혹은 수평 확장등이 있다

</br>

### 실무에서는

- PM2 는 단일 서버 또는 소수 VM/EC2 기반 운영에서 실용적이다
- Nginx + PM2 + Node 조합은 여전히 사용중이다
- 이 환경에서는 PM2 가 프로세스 관리, cluster mode, reload, 로그 파일까지 한 번에 제공하므로 운영 효율이 높다
- 반면 ECS, k8s, Docker orchestrator 가 이미 구축된 환경이라면 역할이 겹칠 수 있다
- 예를 들어 k8s 는 이미 프로세스 격리 대신 컨테이너 하나 == 앱 하나 를 기준으로 health check, restart, rollout, scaling 을 제공한다
- 이때 컨테이너 안에서 다시 PM2 를 운영하면 누가 진짜 생명주기 주체인지 모호해지고 로그 또한 stdout 중심이 아닌 파일 중심으로 틀어지면서 오히려 운영이 복잡해진다
- 즉 인프라 계층이 이미 container orchestrator 환경이라면 PM2 대신 orchestrator 에게 생명주기를 맡기고, 앱은 단일 프로세스로 단순화하는 편이 좋다
