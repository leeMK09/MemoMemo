## Nginx 설정만으로는 연결 수가 늘어나지 않는가

- 서버가 클라이언트 요청을 처리할 때, Nginx 는 단순히 HTTP 요청을 받는 것이 아니라 TCP 연결(socket)을 생성한다
- 이때 리눅스 커널은 이 소켓을 내부적으로 `struct file` 로 관리하며, 각 소켓에는 파일 디스크립터(fd) 가 하나씩 할당된다
  - 즉 OS 는 소켓을 일반 파일과 동일하게 취급하기 때문에, 하나의 네트워크 연결은 하나의 파일 디스크립터를 소비하게 된다

**하나의 요청 흐름**

1. 클라이언트가 TCP SYC 를 보냄
1. 커널이 소켓을 생성
1. 해당 소켓을 프로세스(Nginx worker) 에 전달하면서 fd 를 하나 할당
1. Nginx 는 그 fd 를 기반으로 epoll 에 등록하여 이벤트 기반으로 요청을 처리

이 구조로 인해 연결 수 = fd 사용량이 1:1 로 대응된다

**OS 의 파일 디스크립터 개수 제한**

- OS 는 각 프로세스가 동시에 사용할 수 있는 파일 디스크립터 개수를 제한하고 있는데, 이 제한이 바로 서버가 동시에 처리할 수 있는 연결 수의 근본적인 상한이 된다
- 예를 들어 어떤 프로세스의 파일 디스크립터 제한이 1024 라면, 이 프로세스는 동시에 최대 1024개의 소켓만 열 수 있고, 결국 최대 1024개의 클라이언트 연결만 유지할 수 있게된다
- 이 상황에서 Nginx 는 내부적으로 클라이언트 연결이 들어올 때마다 소켓을 생성하고, 그 소켓을 파일 디스크립터로 관리
- 따라서 Nginx 의 worker 프로세스 하나가 처리할 수 있는 최대 연결 수는 결국 해당 프로세스가 사용할 수 있는 파일 디스크립터 개수에 의해 결정된다

</br>

### 커널 ~ Nginx 요청 흐름

1. 클라이언트의 요청이 들어옴
1. 커널이 SYC 패킷을 받고 listen socket 큐(backlog) 에 넣는다
1. Nginx worker 가 `accept()` 를 호출하면서 새로운 소켓을 가져온다
1. 커널은 이 소켓에 대해 새로운 fd 를 할당한다
1. Nginx 는 해당 fd 를 epoll 에 등록한다 (`epoll_ctl`)
1. 이후 데이터가 도착하면 커널이 `epoll_wait` 을 깨우고
1. Nginx 는 해당 fd 를 기반으로 read/write 를 수행한다

- Nginx 는 연결을 직접 관리하는 것이 아닌 커널이 만들어준 fd 를 받아서 이벤트 루프(epoll) 로 multiplexing 하는 구조
- 따라서 fd 를 더 이상 할당받지 못하면 `accept()` 단계에서 실패, 새로운 연결 자체를 못 받는다

### Nginx worker_connections 설정과 실제 동작의 차이

- Nginx 설정 파일에서 `worker_connections` 라는 값을 설정할 수 있는데, 이 값은 worker 프로세스 하나가 동시에 처리할 수 있는 연결 수를 의미한다
- 이 설정만 보면 마치 이 값을 크게 설정하면 서버가 더 많은 요청을 처리할 수 있는 것처럼 보인다
- 하지만 실제로는 이 값이 단독으로 의미를 가지지 않는다
- 이유는 Nginx 는 운영체제 위에서 동작하는 애플리케이션이기 때문에, 내부적으로 소켓을 생성하려고 할 때마다 운영체제가 허용한 파일 디스크립터 범위 내에서만 자원을 사용할 수 있기 때문이다
- 즉, 만약 운영체제가 한 프로세스 당 1024 개의 파일 디스크립터만 허용하고 있는데 Nginx 설정에서 `worker_connections` 를 10,000 으로 설정해도 실제로는 1024 개 이상 연결을 열 수 없게 된다
- 이 경우 Nginx 는 추가 연결 요청을 받아들이지 못하거나 에러를 발생시키게 된다

**OS 설정이 더 중요한 이유**

- 이 구조에서 중요한 점은 Nginx 설정이 아니라 운영체제 설정이 최종적인 상한을 결정한다는 것이다
- 따라서 서버의 동시 처리량을 늘리고 싶다면 먼저 OS 의 파일 디스크립터 제한을 충분히 크게 설정해야 한다
- 그 이후에 Nginx 의 `worker_connections` 와 `worker_processes` 를 조정해야만 설정이 실제 성능으로 이어지게 된다
  - Nginx 는 단순히 이만큼까지 처리할 수 있도록 준비하겠다는 의도를 표현하는 설정일 뿐이고 실제 자원 할당 권한은 운영체제가 가지고 있는다

**왜 OS 는 fd 를 제한하는가**

- 파일 디스크립터는 단순 숫자가 아니라 커널 내부 리소스를 의미한다
- 각 fd 는 다음을 포함한다
  - socket buffer(send/recv buffer)
  - 커널 메모리 구조체
  - 네트워크 상태 관리 정보 (TCP 상태 머신)
- 즉 fd 가 늘어나면 커널 메모리 사용량이 증가하고 컨텍스트 스위칭 증가 가능성, epoll 이벤트 스캔 비용 증가 비용이 있다
- 그래서 OS 는 기본적으로 제한을 둔다

**worker_connections 를 무작정 올리면**

- `worker_connections` 를 크게 설정하면 epoll 에 등록된 fd 수가 증가하고 이벤트 루프에서 처리해야할 대상이 증가하고 latency 가 증가하게 된다
- 또한 연결 하나당 메모리 사용량이 있기 때문에 수 GB 가 사용될 수 있다

**OS 레벨 튜닝**

- 먼저 프로세스 fd 제한을 먼저 올린다

```json
ulimit -n
/etc/security/limits.conf
fs.file-max
```

- 이 단계에서 커널이 허용하는 fd 상한을 늘린다
- 그 다음 아래와 같이 Nginx 레벨 튜닝을 수행한다

```json
worker_processes auto;
worker_connections 4096;
```

- 실제 최대 연결 수는 `worker_processes * worker_connections`
- 연결 수가 많아지면 TCP 레벨 병목도 생긴다
  - SYN backlog
  - TIME_WAIT 증가
- 혹은 Keep-Alive 를 조절하여 연결 유지 시간이 길면 fd 를 오래 점유할 수 있다
  - 즉 keepalive_timeout 을 줄이면 fd 회전율이 증가한다
- LB 사용, ALB/Nginx 앞단에 부하 분산 처리
  - 한 인스턴스 당 fd 부담 감소, 전체 시스템 처리량 증가
- Event-driven 구조 유지
  - epoll 기반 non-blocking 구조
  - fd 수가 많아도 스레드 폭발이 없음
